# Snug — Database Schema Decision Reference

Every table, every column, every decision explained. This document covers what each column stores, why it exists in the shape it does, and what would break if it were missing or designed differently.

## 1. sessions

**Who writes it:** The Shopify adapter (`@shopify/shopify-app-session-storage-drizzle`) manages this table entirely. You never write to it directly.

**Why it exists:** Shopify embedded apps cannot use browser cookies for authentication because Safari and Chrome block third-party cookies. Instead, Shopify issues short-lived signed JWTs called session tokens. The adapter stores the merchant's Shopify access token here after OAuth completes, and looks it up on every authenticated request to the dashboard.

| Column | Type | Decision |
|--------|------|----------|
| `id` | text PK | Shopify-generated session identifier. Text not UUID because Shopify controls the format. |
| `shop` | text | The merchant's myshopify.com domain. This is the stable identifier Shopify uses for a store — it never changes even if the merchant changes their storefront domain. |
| `state` | text | OAuth state parameter used to prevent CSRF attacks during the install flow. |
| `isOnline` | boolean | Whether this is an online token (tied to a specific staff member's browser session) or an offline token (permanent, used for background jobs). Snug uses offline tokens. |
| `scope` | text | The permission scopes granted by the merchant during install. Stored so you can detect if scopes have changed and prompt re-authorisation. |
| `expires` | timestamp | When this session expires. Online sessions expire quickly. Offline sessions are long-lived. |
| `accessToken` | text | The actual Shopify API access token. This is what you use to call the Shopify Admin API on behalf of the merchant. Treat this as a secret — never log it, never expose it in responses. |
| `userId` | integer | The Shopify staff member ID for online sessions. Null for offline sessions. |
| `firstName`, `lastName`, `email` | text | Staff member details for online sessions. Null for offline sessions. |
| `accountOwner` | boolean | Whether this session belongs to the store owner. Relevant if you ever restrict certain dashboard features to account owners only. |
| `locale` | text | The staff member's preferred language. Useful for future internationalisation of the dashboard. |
| `collaborator` | boolean | Whether the authenticated user is a Shopify Partner collaborator rather than a store staff member. |
| `emailVerified` | boolean | Whether the staff member's email has been verified with Shopify. |
| `refreshToken` | text | Used by the `expiringOfflineAccessTokens` future flag we enabled. Allows silent token refresh when the access token expires without forcing a re-install. |
| `refreshTokenExpires` | timestamp | When the refresh token itself expires. After this point a full re-install is required. |

**Why you should not touch this table:** The adapter owns the schema contract. If you add columns, rename columns, or change types, the adapter will break silently because it writes and reads using hardcoded column names.

## 2. organizations

**Who writes it:** The OAuth callback handler in your Remix app, immediately after a merchant completes the install flow.

**Why it exists:** This is the root record for every merchant. Every other merchant-owned table references this table. When you need to know anything about a merchant — their plan, their brand, whether their widget is live — you start here.

| Column | Type | Decision |
|--------|------|----------|
| `org_id` | uuid PK | UUID generated by the database, not by application code. Using a database-generated UUID means you never have to worry about ID collisions even across concurrent installs. |
| `shop_domain` | text UNIQUE | The merchant's myshopify.com domain. This is the join key between this table and the `sessions` table. It is unique because one store maps to exactly one organisation. If you used a numeric Shopify store ID instead, you would need to look it up from the session on every request — the domain is already in the session and in every request header. |
| `brand_slug` | text | The merchant's own brand name in lowercase slugified form, e.g. `snitch`, `bewakoof`. This must match the `brand` column in `brand_size_charts` exactly. It is stored as a slug rather than a display name to prevent case mismatch bugs — if a merchant typed `Snitch` and the database has `snitch`, the algorithm lookup would silently fail. The application layer is responsible for lowercasing and slugifying on write. |
| `plan_tier` | text | One of `free`, `starter`, `pro`. Controls request rate limits and feature access. Stored here rather than derived from a billing system because at v0 plan changes are manual. When you integrate Shopify Billing API, this column gets updated by the billing webhook handler. |
| `onboarding_complete` | boolean | Whether the merchant has finished the initial setup flow — configured their brand slug and activated the widget. This is separate from `widget_active` because a merchant can complete onboarding but then turn the widget off temporarily. Without this split you cannot tell the difference between "never finished setup" and "deliberately paused". |
| `widget_active` | boolean | Whether the widget is currently serving recommendations on the merchant's storefront. The Worker checks this flag on every request via the KV entry for this shop domain. When false, the Worker returns a 403 without running the algorithm. |
| `usage_remaining` | integer | How many predictions this merchant has left in their current billing period. Set to the plan limit when the billing period starts and kept current by the cron job. The Worker reads this value from KV — where it is mirrored — to enforce the monthly cap on every request. Never written by the Worker directly. |
| `billing_period_start` | timestamp | When the current billing period began. Used by the cron job to know which `usage_logs` rows to count when recomputing `usage_remaining`. Resets to the current timestamp when a billing period rolls over or when a merchant upgrades their plan. |
| `created_at` | timestamp | When the merchant installed. Used for cohort analysis — are newer merchants retaining better than older ones? |
| `updated_at` | timestamp | When any field on this row was last changed. Updated automatically on every write via application logic. |

**Why `is_active` was split into two columns:** The original design had a single `is_active` flag. This was wrong because it conflated two independent states. A merchant who has not finished onboarding is not the same as a merchant who completed onboarding but paused their widget. The dashboard onboarding flow needs to know which screen to show. The Worker needs to know whether to serve recommendations. These are different questions requiring different answers.

**Why `usage_remaining` lives here and not in Redis or a separate table:** Usage limits in Snug are monthly caps — a merchant on the starter plan gets 1000 predictions per billing period, not 1000 per minute. This is fundamentally different from a rate limit and needs different infrastructure. Redis is wrong for this because Redis is ephemeral — it is designed for short-lived counters that expire, not for state that must persist for 30 days and feed billing calculations. A separate table is wrong because it adds a join on every request for data that belongs to the merchant record anyway.

`usage_remaining` on the organizations row is the right home because it is merchant state, it lives next to the plan tier that determines the limit, and it is already being pushed to KV as part of the org record that the Worker reads on every request. The Worker reads one KV entry and gets both the plan tier and the remaining count — no second lookup needed.

The column is never written by the Worker directly. Writing it on every request would require the Worker to have UPDATE access to the organizations table, which violates the least-privilege model, and would create write contention on the most-read table in the system under concurrent load. Instead the cron job maintains it: it counts `usage_logs` rows for the current billing period, subtracts from the plan limit, and writes the result back here and to KV. The slight lag between a request being served and `usage_remaining` being decremented is acceptable — you are not billing to the millisecond.

Redis is still used alongside this, but only for burst protection — preventing a merchant from firing 1000 requests in the first second of their billing period before the cron job has caught up. Redis handles short-term spikes. The organizations column handles the long-term monthly cap. They are complementary.

## 3. widget_configs

**Who writes it:** The merchant via the dashboard. One row is created automatically when the organisation is created, populated with defaults.

**Why it exists:** Merchants need to control how the widget looks and behaves on their storefront without editing code. This table is the configuration layer between the merchant's preferences and the widget renderer.

| Column | Type | Decision |
|--------|------|----------|
| `config_id` | uuid PK | Standard primary key. |
| `org_id` | uuid FK → organizations | One-to-one with organizations. Every merchant has exactly one widget config. If you ever need per-product widget configs, that is a separate table — this one stays at the org level. |
| `position` | text | Where the widget renders on the product page. One of `below_add_to_cart`, `above_add_to_cart`, `below_price`. Stored as an explicit column rather than inside the JSONB config because the widget loader JavaScript needs to read this value to know where in the DOM to inject itself. It is a structural concern, not a visual one. |
| `is_enabled` | boolean | Whether the widget renders at all. This is the merchant-facing on/off toggle. Distinct from `widget_active` on `organizations` — `is_enabled` is the merchant's deliberate choice, `widget_active` could also be false because onboarding is incomplete. The Worker checks both before serving. |
| `config` | jsonb | All visual and behavioural settings that are opaque to the database — primary colour, button label text, font size overrides, animation preferences. Stored as JSONB because this set of settings will grow over time without requiring schema migrations. The widget renderer reads the entire blob and applies it. The database never needs to filter or query individual fields within it. |
| `created_at` | timestamp | When this config was created. |
| `updated_at` | timestamp | When the merchant last changed their widget settings. |

**Why visual settings are in JSONB but position and is_enabled are explicit columns:** The rule is: anything the database or application infrastructure needs to query or branch on gets an explicit column. Anything that is only read by the widget renderer as a blob goes in JSONB. The database needs to know `position` to generate the widget embed snippet. The database needs to know `is_enabled` to decide whether to serve the widget. The database does not need to know the button colour.

## 4. fit_size_charts

**Who writes it:** The merchant via the dashboard or via the UI Extension in the Shopify product editor. The application backend also writes `body_min_cm` and `body_max_cm` automatically when the merchant saves a row.

**Why it exists:** This is the merchant's own sizing data. It is the target side of every prediction. When a shopper says "I am a Medium in Uniqlo, what size am I in this brand?", the algorithm needs this table to answer the second half of that question.

**The fundamental structure decision — one row per size label, not one row per chart:** An earlier design stored all sizes for a garment type in a single JSONB column. That was wrong because it meant the algorithm had to deserialise the entire JSON blob to look up a single size, and Postgres could not do indexed lookups on individual size data. The current design has one row per `(org_id, garment_type, size_label)` combination, which means a query for "what are the chest measurements for org X's hoodie size L?" is a single indexed point lookup.

| Column | Type | Decision |
|--------|------|----------|
| `chart_id` | uuid PK | Row identifier. |
| `org_id` | uuid FK → organizations | Which merchant owns this row. |
| `garment_type` | text CHECK | One of `tshirt`, `shirt`, `polo`, `sweatshirt`, `hoodie`, `jacket`, `kurta`, `top`. Enforced as a check constraint at the database level, not just in application code. A typo in the dashboard form cannot corrupt this column. |
| `size_label` | text | The size as the merchant labels it — `S`, `M`, `L`, `XL`, `38`, `40`, etc. Text not an enum because merchants use different labelling systems. Some use alphabetic sizes, some use numeric chest sizes, some use regional sizing numbers. The algorithm matches by value, not by position in an ordered list. |
| `fit_type` | text CHECK | One of `slim`, `regular`, `oversized`. Enforced as a check constraint. The algorithm uses this to compute the fit type match signal in the confidence score. A cross-fit prediction (slim reference brand, oversized target) incurs a confidence penalty. |
| `chest_min_cm` | numeric NOT NULL | The lower bound of the garment's chest measurement in centimetres. Always stored in centimetres regardless of what unit the merchant enters — the application layer converts on write. NOT NULL because chest is the primary matching axis. Without it the algorithm cannot run. |
| `chest_max_cm` | numeric NOT NULL | The upper bound. The algorithm uses the midpoint of min and max as its best estimate of where within the range the actual garment measures. The width of the range (max minus min) feeds the range tightness signal in the confidence score — a narrow range produces a higher confidence prediction. |
| `length_min_cm` | numeric nullable | Garment body length lower bound. Nullable because not all merchants publish length. When present, it is used as a secondary signal in confidence scoring for garment types where length matters significantly (kurtas especially). When absent, the algorithm degrades gracefully and uses chest alone. |
| `length_max_cm` | numeric nullable | Garment body length upper bound. |
| `shoulder_min_cm` | numeric nullable | Shoulder width lower bound. Nullable. Most relevant for structured garments like jackets and shirts. |
| `shoulder_max_cm` | numeric nullable | Shoulder width upper bound. |
| `ease_value_cm` | numeric NOT NULL | How much extra fabric this garment has beyond the wearer's body measurement. This is the bridge between garment measurement space and body measurement space. The algorithm subtracts this from the garment midpoint to recover the body anchor. NOT NULL because ease is required for the core arithmetic. |
| `ease_source` | text CHECK | One of `explicit`, `inferred`, `user_calibrated`. Enforced as a check constraint. This feeds directly into the ease source trust signal in the confidence score. `explicit` means the merchant told you the ease value directly. `inferred` means you computed it from population averages. `user_calibrated` means you derived it from accumulated user fit declarations. Each has a different trust multiplier in the confidence calculation. |
| `body_min_cm` | numeric | Precomputed lower bound of the wearer's body chest measurement corresponding to this size. Computed as `chest_min_cm minus ease_value_cm`. Written by the application backend automatically when the merchant saves the row. This is what the inference engine reads at query time — the body anchor arithmetic happens once at write time, not on every prediction request. |
| `body_max_cm` | numeric | Precomputed upper bound. Computed as `chest_max_cm minus ease_value_cm`. |
| `extra_measurements` | jsonb | Waist, hip, sleeve, or any other measurement the merchant wants to store beyond the standard columns. The algorithm does not use this in v0 — it is display-only in the widget. It exists so merchants can store their complete size chart data in one place rather than having to maintain it separately. Stored as JSONB because the set of custom measurements varies by merchant and garment type. |
| `created_at` | timestamp | When this size row was created. |
| `updated_at` | timestamp | When this row was last edited. If `updated_at` is more recent than the `body_min_cm` and `body_max_cm` were last written, the application knows it needs to recompute the body ranges. |

**The unique constraint on `(org_id, garment_type, size_label)`:** A merchant cannot have two rows for the same garment type and size label. Without this constraint, a merchant who accidentally creates a duplicate hoodie size M row would produce two candidates for the algorithm to match against, making prediction non-deterministic. The constraint makes this a hard error at the database level.

**Why `body_min_cm` and `body_max_cm` are stored rather than computed at query time:** The ease subtraction is simple arithmetic but it involves ease values that were themselves inferred from population averages — a non-deterministic step. Computing it once at write time means the inference engine is fully deterministic at query time. It also means the Worker reads one row and has everything it needs — no additional computation, no risk of the result changing between requests.

## 5. garment_mappings

**Who writes it:** The Shopify product sync job and the UI Extension in the Shopify product editor.

**Why it exists:** The Widget needs to know, when a shopper is on a specific product page, which garment type that product is. Shopify product metadata does not reliably encode garment type. This table is the explicit mapping between a Shopify product ID and the garment type used to look up the merchant's size chart.

| Column | Type | Decision |
|--------|------|----------|
| `mapping_id` | uuid PK | Row identifier. |
| `org_id` | uuid FK → organizations | Which merchant owns this mapping. |
| `shopify_product_id` | text | The Shopify GID for the product, e.g. `gid://shopify/Product/8234567890`. Text not integer because Shopify GIDs are strings. This is the stable identifier — it never changes even if the product title or handle changes. |
| `garment_type` | text CHECK | The garment type this product maps to. The Widget reads this and uses it with `org_id` to look up the right rows in `fit_size_charts`. |
| `created_at` | timestamp | When this mapping was created. |
| `updated_at` | timestamp | When the mapping was last changed — for example if the merchant reclassified a product from `tshirt` to `polo`. |

**Why `chart_id` was removed:** An earlier design stored a direct foreign key to a specific `fit_size_charts` row. This meant if the merchant updated their size chart, every product mapping would still point at the old chart until manually updated. The current design removes `chart_id` entirely. The lookup path is `shopify_product_id → garment_type + org_id → fit_size_charts WHERE org_id = x AND garment_type = y`. When the merchant updates a size chart, all products mapped to that garment type automatically reflect the new data. No cascading updates needed.

**Why `product_title` was removed:** Product titles change in Shopify. Storing them here creates stale data — the merchant renames their product in Shopify and the dashboard shows the old name. Since we have `read_products` scope, the dashboard fetches the current title from the Shopify API at render time. The stable identifier `shopify_product_id` is all we need to store.

**Why `metafields` JSONB was removed:** The original design included a catch-all JSONB column for product metafields. It was removed because we had no concrete plan for what would go in it. JSONB columns added speculatively become archaeological mysteries — six months later nobody remembers what was supposed to be stored there or in what shape. When there is a specific need for product metafield storage, add a specific column for it.

## 6. usage_logs

**Who writes it:** The Cloudflare Worker, via `ctx.waitUntil()` after every prediction response is sent. This write is non-blocking — the shopper receives their response before the database write completes.

**Why it exists:** Two purposes. First, it is the metering table — the billing cron job reads it to count recommendations served per merchant per billing period. Second, it is the analytics table — the merchant dashboard reads it to show recommendation volume, confidence distribution, and which sizes are most commonly predicted.

| Column | Type | Decision |
|--------|------|----------|
| `log_id` | uuid PK | Row identifier. |
| `org_id` | uuid | Which merchant's widget served this prediction. Not a foreign key constraint because the Worker writes this table directly via a restricted database user that only has INSERT permission on this table. A FK constraint would require the Worker to have read access to the `organizations` table to validate the reference, which violates the principle of least privilege. The application layer validates `org_id` values — the database trusts the Worker to write correct data. |
| `ref_brand` | text | The brand the shopper said they own. Stored for analytics — which reference brands are shoppers using most? This data can inform which brands to prioritise for scraping. |
| `ref_garment` | text | The garment type the shopper selected. |
| `ref_size` | text | The size label the shopper entered for their reference brand garment. |
| `predicted_size` | text | The size the algorithm recommended for the target brand. |
| `confidence` | integer | The confidence score from 0 to 100. Stored as an integer. The confidence label (`high`, `medium`, `low`) is derived from this value in application code — storing both would be redundant and would create a risk of them being out of sync if confidence thresholds change. |
| `response_ms` | integer | How long the prediction took in milliseconds. Used to monitor Worker performance over time. |
| `created_at` | timestamp | When this prediction was served. The primary dimension for all time-series analytics. |

**Why `returned`, `return_reason`, and `shopify_order_id` were removed:** These columns were in an earlier design to track whether a purchase made using a size recommendation was later returned. They were removed for v0 because none of them can be populated at write time. The recommendation happens before any order exists. Order and return data comes from Shopify webhooks later and requires a separate correlation job to link it back to a specific prediction. Storing three nullable columns that will be null on every row at write time adds confusion without value. When return tracking is built, it will be a separate `return_events` table that references `usage_log_id`.

## 7. brand_size_charts

**Who writes it:** The scraper pipeline. The Remix app and the Cloudflare Worker never write to this table.

**Why it exists:** This is the reference data for the shopper's side of every prediction. When a shopper says "I am a Medium in Uniqlo", the algorithm looks up Uniqlo's medium chest range in this table to establish the body anchor. Without this table the algorithm has nothing to compare against.

| Column | Type | Decision |
|--------|------|----------|
| `brand` | text | Brand name in lowercase slugified form, e.g. `uniqlo`, `zara`, `snitch`. Lowercase is enforced by the scraper on write. The Worker lowercases incoming `ref_brand` values before lookup, so case mismatches cannot cause silent failures. |
| `garment_type` | text CHECK | Same enum as `fit_size_charts`. Enforced as a check constraint so scraper typos cannot corrupt the column. |
| `size_label` | text | The size as printed on the garment. Text because different brands use different size systems — alphabetic, numeric, regional. |
| `chest_min_cm` | numeric NOT NULL | Lower bound of the garment chest measurement in centimetres. The scraper converts all measurements to centimetres before writing, regardless of what unit the brand published. The algorithm never deals with inches or other units. |
| `chest_max_cm` | numeric NOT NULL | Upper bound. |
| `length_min_cm` | numeric nullable | Garment body length lower bound. Nullable because not all brands publish this. |
| `length_max_cm` | numeric nullable | Garment body length upper bound. |
| `shoulder_min_cm` | numeric nullable | Shoulder width lower bound. |
| `shoulder_max_cm` | numeric nullable | Shoulder width upper bound. |
| `fit_type` | text CHECK | Enforced check constraint. The fit type of the reference brand's garment feeds into the cross-fit penalty calculation in the confidence score. |
| `ease_value_cm` | numeric NOT NULL | The ease associated with this size row. Determined by one of three sources in order of preference: explicit (brand states it), inferred (computed from population average body measurements), user calibrated (derived from accumulated user fit declarations). |
| `ease_source` | text CHECK | Which of the three sources produced the ease value. Enforced check constraint. Feeds directly into the ease source trust signal in the confidence score. `explicit` has trust 1.0, `user_calibrated` 0.85, `inferred` 0.75. |
| `scraped_at` | timestamp | When the scraper last wrote this row. Feeds the data freshness signal in the confidence score. Data scraped within 30 days scores 1.0. Beyond 180 days scores 0.35. The algorithm uses the older of the reference and target scrape timestamps as the freshness input — the weakest link determines the ceiling. |

**The unique constraint on `(brand, garment_type, size_label)`:** Prevents the scraper from creating duplicate rows on re-runs. The scraper uses upsert semantics — insert or update on conflict — so a re-scrape updates `scraped_at` and any changed measurements rather than creating a second row.

## 8. anthropometric_anchors

**Who writes it:** Seeded once at database setup from NIFT (National Institute of Fashion Technology, India) survey data. Never updated at runtime.

**Why it exists:** When the scraper cannot find explicit ease values from a brand's website, it infers ease by comparing the garment's midpoint measurement against the population average body measurement for that size. This table provides those population averages. Without it, Source B ease inference cannot run and the scraper falls back to a hardcoded default, which is less accurate.

| Column | Type | Decision |
|--------|------|----------|
| `gender` | text CHECK | One of `M`, `F`, `unisex`. Enforced as a check constraint. The India-specific NIFT data is primarily calibrated for male sizing, which is the dominant use case for the Indian D2C brands in the initial target market. Female and unisex rows can be added when the algorithm is extended for women's topwear. |
| `size_label` | text | Canonical size bucket — `XS`, `S`, `M`, `L`, `XL`, `XXL`. These are standard labels that the scraper maps brand-specific size labels onto for the purpose of ease inference. A brand that uses `38` for what most brands call `M` gets mapped to `M` for this lookup. |
| `avg_chest_cm` | numeric | Population average chest circumference for this gender and size bucket. The NIFT male values seeded at setup: XS=82, S=86, M=91, L=96, XL=102, XXL=108. These are treated as constants — they do not change at runtime. |
| `source` | text | Where the data came from. `NIFT_2020` for the initial seed. If a future survey produces better data, a new source value is added and the scraper can be updated to prefer it. Keeping the source column makes it possible to audit which population data drove which ease inferences. |

**Why India-specific data:** The primary target market for v0 is Indian D2C brands selling to Indian customers. Using NIFT data rather than international sizing survey data produces more accurate ease inferences for this market. When the product expands to other markets, additional rows with country-specific sources can be added.

## 9. brand_requests

**Who writes it:** The merchant via the dashboard, when they search for their brand during onboarding and it is not found in `brand_size_charts`.

**Why it exists:** The scraper only covers brands that are already in the database. When a new merchant installs and their brand is not yet scraped, they cannot use the widget. This table is the queue that tells the Snug team which brands to add next. Without it, merchants whose brands are missing have no recourse and no visibility into when the problem will be resolved.

| Column | Type | Decision |
|--------|------|----------|
| `request_id` | uuid PK | Row identifier. |
| `org_id` | uuid FK → organizations | Which merchant submitted this request. Allows the system to notify the merchant when their brand is added and to deduplicate — if two merchants request the same brand, the team can see that and prioritise accordingly. |
| `brand_name` | text | The brand name as the merchant typed it. Not slugified because you want to see exactly what they entered — it might reveal spelling variations or regional naming differences that the scraper needs to account for. |
| `brand_url` | text nullable | The brand's website, optionally provided by the merchant. The scraper uses this as the starting point rather than having to discover the URL from a search. Nullable because merchants may not know the URL off the top of their head. |
| `status` | text CHECK | One of `pending`, `in_progress`, `completed`. Enforced as a check constraint. Displayed in the merchant's dashboard so they can see whether their request has been picked up. Updated manually by the Snug team or automatically when the scraper successfully writes rows for the brand. |
| `created_at` | timestamp | When the request was submitted. Used to prioritise — older requests that have not been fulfilled should be actioned before newer ones. |
| `updated_at` | timestamp | When the status was last changed. |

## 10. scrape_runs

**Who writes it:** The scraper pipeline, once per brand per scrape run.

**Why it exists:** Without this table there is no operational visibility into the scraper. If the scraper silently fails for a brand, you will not know until a merchant complains that their predictions have low confidence scores because the data is stale. This table is the health log for the scraper system.

| Column | Type | Decision |
|--------|------|----------|
| `run_id` | uuid PK | Row identifier. |
| `brand` | text | Which brand this run attempted to scrape. Matches the `brand` column in `brand_size_charts`. |
| `status` | text CHECK | One of `success`, `partial`, `failed`. `partial` means some garment types were scraped but not all. Enforced as a check constraint. |
| `rows_written` | integer | How many rows were written to `brand_size_charts` in this run. A run that reports `success` but wrote zero rows is suspicious and should alert. |
| `error_message` | text nullable | If status is `failed` or `partial`, the error message or exception that caused it. Nullable because successful runs have no error. |
| `started_at` | timestamp | When the scraper began processing this brand. |
| `completed_at` | timestamp | When the scraper finished. The difference between `started_at` and `completed_at` gives scrape duration — useful for detecting when a brand's website is slowing down the scraper. |

**Why this table feeds the freshness signal indirectly:** The confidence score's freshness signal reads `scraped_at` from `brand_size_charts`, not from `scrape_runs`. But `scrape_runs` tells you operationally why `scraped_at` might be stale — was the last scrape a failure? How long ago did it run? This is the difference between user-facing confidence data and internal operational monitoring.

## 11. Cross-cutting decisions

### All primary keys are UUIDs

Integer auto-increment IDs are predictable. A merchant who knows their `org_id` is `42` can guess that `41` and `43` are also valid merchants. UUID primary keys are not guessable. This matters most for `org_id` because it appears in the KV store and in Worker requests.

### All timestamps are stored in UTC

Postgres stores timestamps without timezone information by default. The application layer is responsible for ensuring all timestamps written to the database are UTC. The dashboard converts to the merchant's local timezone for display. Storing in local time would make cross-merchant analytics and scraper scheduling impossible.

### Check constraints on all enums

Every column with a restricted set of values — `garment_type`, `fit_type`, `ease_source`, `plan_tier`, `status`, `gender` — has a Postgres check constraint enforcing the allowed values. This is in addition to application-level validation. The reason is defence in depth: the scraper, the Worker, and the dashboard are three different codebases that could each make independent mistakes. A check constraint is the last line of defence that catches corruption regardless of which system caused it.

### All measurements in centimetres

Brands publish measurements in different units — centimetres, inches, sometimes both. All measurements are converted to centimetres before being written to the database. The algorithm works entirely in centimetres. The widget converts to the shopper's preferred unit for display if needed. Mixing units in the database would make every comparison and calculation error-prone.

### JSONB used sparingly and only for opaque data

JSONB is used in two places: `widget_configs.config` for visual styling settings, and `fit_size_charts.extra_measurements` for custom merchant measurements. In both cases the database never needs to filter or query individual fields within the blob — the data is always read as a whole unit and processed in application code. Any column that the database, the algorithm, or the Worker needs to query or branch on is an explicit typed column, not buried in JSONB.

### The Worker writes `usage_logs` with a restricted database user

The Cloudflare Worker has a Neon connection string that grants only `INSERT` on `usage_logs`. It cannot read, update, or delete any row in any table. If the Worker's credentials are ever compromised, the attacker can write garbage log rows but cannot read merchant data, access tokens, or size chart information. The Remix dashboard app has a separate connection string with full access to all tables.

### Rate limiting and usage cap enforcement are two separate systems

Snug has two distinct limits that are often confused but require completely different infrastructure.

**Burst protection** is a short-term limit — no merchant can fire more than N requests per minute regardless of their plan. This prevents a single merchant from flooding the Worker during a flash sale and degrading performance for everyone else. Redis handles this via atomic INCR on a per-minute key that expires after 120 seconds. Redis is the right tool because the operation must be atomic (two simultaneous requests cannot both read the same count and both pass), it must be fast (sub-10ms on the critical path), and the data is genuinely ephemeral — minute-level counters have no value after they expire.

**Monthly usage caps** are long-term limits — a merchant on the starter plan gets 1000 predictions per billing period. This is a business constraint that drives billing and must be accurate and persistent. `usage_remaining` on the organizations row is the source of truth, maintained by a cron job that reads `usage_logs` and writes back the current remaining count. KV mirrors this value so the Worker can read it in sub-millisecond without a Neon query on the critical path.

The two systems work in sequence on every request: Redis burst check first, monthly cap check second. Passing both is required to proceed.

**The cron job responsibility:** Runs on Railway every few minutes. For each active merchant it counts `usage_logs` rows where `created_at >= billing_period_start`, computes `plan_limit - count`, writes that number to `organizations.usage_remaining` in Neon, and writes the same value to `KV usage:{org_id}`. When a billing period rolls over, it resets `usage_remaining` to the full plan limit and updates `billing_period_start`.

### `brand_size_charts` and `anthropometric_anchors` have no foreign key relationships

These are scraper-owned tables. They exist independently of the merchant data graph. There is no FK from `organizations.brand_slug` to `brand_size_charts.brand` — a soft reference enforced by the application layer rather than a hard constraint — because the scraper can rename or re-slug a brand at any time, and a FK constraint would cascade that change into the merchant table in unpredictable ways. The application layer validates that `brand_slug` exists in `brand_size_charts` when a merchant sets it, but the database itself does not enforce this at the constraint level.
